# Copilot Instructions: Local Development LLM Project

## Project Overview

This repository sets up a **local LLM for development assistance** as a GitHub Copilot alternative, running in Docker on WSL2 Ubuntu.

**Purpose**: Privacy-first, cost-free, offline code assistance using GPU-accelerated local language models.

---

## System Context

### Hardware Specifications

- **Platform**: Windows 10 Pro with WSL2 (Ubuntu)
- **CPU**: Intel Core i7-14700KF (20 cores, 28 threads, high-end)
- **RAM**: 32GB DDR5
- **GPU**: NVIDIA RTX 3060 12GB GDDR6
  - Supports 7B models easily
  - Can run 14B-16B models with 4-bit quantization
  - Cannot run 32B+ models

### Technology Stack

- **LLM Runtime**: Ollama (containerized)
- **Container Platform**: Docker with NVIDIA GPU passthrough
- **VS Code Integration**: Continue.dev extension
- **Primary Model**: Qwen3-Coder (7B or 14B variants)
- **Operating System**: WSL2 Ubuntu (optimal for GPU + Docker)

---

## Key Technical Decisions

### 1. Why Ollama?

- **Simple model management**: Pull, run, switch models with single commands
- **Automatic optimization**: Handles quantization and GPU allocation
- **OpenAI-compatible API**: Standard REST endpoints
- **Extensive model library**: 100+ pre-configured models
- **Low resource overhead**: Efficient C++ implementation

### 2. Why Docker?

- **Isolation**: Keep LLM environment separate from host
- **Reproducibility**: Easy to backup, clone, or migrate
- **GPU Passthrough**: Works seamlessly with nvidia-container-toolkit
- **Version Control**: Easy rollback and updates

### 3. Why WSL2?

- **Near-native GPU performance**: 98-99% of Linux GPU performance
- **Better Docker integration**: More efficient than Docker Desktop on Windows
- **Linux tooling**: Superior ecosystem for LLM tools
- **File I/O**: Excellent performance on WSL filesystem (`/home/*`)

⚠️ **IMPORTANT**: Files must stay on WSL filesystem (`/home/joscha/...`), not Windows mount (`/mnt/c/...`), for optimal performance.

### 4. Why Qwen3-Coder?

- **Best-in-class** coding performance for size (latest generation)
- **92+ languages** supported
- **Fast inference** on available hardware
- **Active development** by Alibaba's Qwen team
- **Strong benchmarks** on HumanEval, MBPP, MultiPL-E
- **Improved** over Qwen2.5 with better instruction following

## Important Commands

### Docker Management

```bash
# Start Ollama container
docker-compose up -d

# Stop Ollama container
docker-compose down

# View logs
docker-compose logs -f ollama

# Restart container
docker-compose restart
```

### Model Management

```bash
# Pull a new model (Qwen3-Coder 7B)
docker exec -it ollama ollama pull qwen3-coder:7b

# Pull 14B variant (uses more VRAM)
docker exec -it ollama ollama pull qwen3-coder:14b

# List installed models
docker exec -it ollama ollama list

# Remove a model to save space
docker exec -it ollama ollama rm <model-name>

# Run model interactively (for testing)
docker exec -it ollama ollama run qwen2.5-coder:7b
```

### Alternative Models to Try

```bash
# DeepSeek Coder V2 (excellent alternative)
docker exec -it ollama ollama pull deepseek-coder-v2:16b

# CodeLlama (Meta's model)
docker exec -it ollama ollama pull codellama:7b

# CodeGemma (Google's model)
docker exec -it ollama ollama pull codegemma:7b

# Phi-3.5 (Microsoft's efficient model)
docker exec -it ollama ollama pull phi3.5:latest
```

### GPU Monitoring

```bash
# Check GPU usage (run on host Windows or WSL)
nvidia-smi

# Monitor GPU usage continuously
watch -n 1 nvidia-smi

# Inside Docker
docker exec -it ollama nvidia-smi
```

## Common Tasks for AI Agents

### When User Wants to Switch Models

1. **Pull the new model**:

   ```bash
   docker exec -it ollama ollama pull <new-model>
   ```

2. **Update Continue extension config**:
   - Edit VS Code settings
   - Change model name in Ollama provider config

3. **Optional: Remove old model** to save disk space:
   ```bash
   docker exec -it ollama ollama rm <old-model>
   ```

### When User Reports Performance Issues

1. **Check GPU usage**:
   ```bash
   nvidia-smi
   ```
2. **Check if model fits in VRAM**:
   - 7B models: Should use ~6-8GB
   - 14B models: Should use ~10-12GB
   - If VRAM is maxed out → suggest smaller model or quantization

3. **Check Docker resources**:

   ```bash
   docker stats ollama
   ```

4. **Verify GPU passthrough**:
   ```bash
   docker exec -it ollama nvidia-smi
   ```

### When Adding New Features

**Principles**:

- Keep it simple: Ollama handles complexity
- Use Docker volumes for persistence
- Don't store models in git (too large)
- Document model compatibility
- Test on 7B model first before recommending larger models

## Troubleshooting Guide for AI Agents

### Issue: "Model runs slow"

**Diagnose**:

1. Check if model is using GPU: `docker exec -it ollama nvidia-smi`
2. Check model size vs VRAM
3. Verify no other GPU processes running

**Solutions**:

- Switch to smaller model (14B → 7B)
- Close other GPU applications
- Check Docker has GPU access

### Issue: "Out of VRAM"

**Solutions**:

- Switch to 7B model
- Close other applications using GPU
- Use more aggressive quantization (q4 variant)

### Issue: "Container won't start"

**Diagnose**:

1. Check Docker daemon: `docker ps`
2. Check nvidia-container-toolkit: `docker run --rm --gpus all nvidia/cuda:12.6.0-base-ubuntu22.04 nvidia-smi`

**Solutions**:

- Restart Docker daemon
- Reinstall nvidia-container-toolkit
- Check Docker compose file syntax

### Issue: "Continue extension can't connect"

**Diagnose**:

1. Check Ollama running: `docker ps | grep ollama`
2. Check port 11434: `curl http://localhost:11434/api/tags`

**Solutions**:

- Restart Ollama container
- Verify port mapping in docker-compose.yml
- Check firewall settings
